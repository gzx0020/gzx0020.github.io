FastKV 论文阅读笔记

1. 论文在解决什么问题？

大语言模型（LLM）处理长文本（比如128K tokens）时，有一个很大的瓶颈：KV Cache。 - KV Cache是什么？ 为了在生成每个新token时不用重新计算所有历史token的信息，Transformer的Attention层会把之前所有token的Key和Value缓存下来。(可以读我之前写的注意力机制) - 问题在哪？ 文本越长，KV Cache，导致： 1. 内存不够用：例如，处理128K tokens的文本，KV Cache可能比模型本身还占内存。 2. 速度变慢：尤其是在“预填充”阶段（Prefill Stage，即处理用户输入的整个提示词），需要为所有输入token计算并存储KV Cache，计算量和延迟很高。TTFT（首token时间）是影响用户体验的关键指标。

现有的KV Cache压缩方法，比如SnapKV，主要在生成阶段节省内存，但对预填充阶段的加速没什么帮助。而另一个方法GemFilter虽然能加速预填充，但因为它直接删掉了输入提示词中的部分token，导致模型丢失了全局上下文信息，精度下降。

所以，这篇论文的目标就是：找到一个既能大幅加速预填充阶段（降低TTFT）、又能保持模型精度的KV Cache压缩方法。 - 补充：什么是TTFT？

TTFT 的全称是 Time To First Token，指的是从你按下送，把完整的Prompt交给大模型开始，到大模型生成的第一个token为止所花费的时间。

2. 核心方法：FastKV 与 Token-Selective Propagation (TSP)

FastKV 的核心方法是 Token-Selective Propagation (TSP) 机制。

核心思想

  1. 早期层（TSP层之前）：让模型像正常一样，完整地处理所有输入token。这样，每个token都经过了多层Attention的“信息融合”，已经包含了丰富的上下文信息。
  2. 在某个中间层（TSP层）：从所有token中，筛选出一小部分“最重要”的token。
  3. 后期层（TSP层之后）：只对这一小部分筛选出来的token进行后续计算。因为token数量大大减少，后续层的计算量也大幅降低，从而实现了加速。

$S_i^{l,g}=\frac{1}{H_G}\sum_{h=h_g}^{h_g+H_G}S_i^{j,h}$

( S_i^{l,g} ) 第 l 层、第 g 个注意力Attention Head Group中，第 i 个 token 的组重要性得分。这是该公式的计算结果，代表一个Attention Head Group对某个 token 的注意力得分。

( H_G ) 每个注意力Attention Head Group中包含的 Attention Head 数量。这是一个预设的超参数，用于控制组的规模。

( h_g ) 第 g 个组的起始 Attention Head 索引。例如，如果每组有8个头，那么第一组 ( h_g = 0 )，第二组 ( h_g = 8 )，以此类推。

( h_g + H_G ) 第 g 个组的结束 Attention Head 索引。即对一个完整组内所有头进行遍历。

( S_i^{j,h} ) 第 j 层、第 h 个 Attention Head 中，第 i 个 token 的重要性得分。

  • 这里插入一个和SnapKV的对比 SnapKV 为每个头单独计算和选择， 每个头选自己的token，不同头保留的token集合可能完全不同

$S_i^{l,h}=\frac{1}{2w_p+1}\sum_{m=-w_p}^{w_p}\sum_{n=0}^{N_\mathrm{obs}}Att_l[h,N_I-n,i+m]$

FastKV 按组统一计算和选择， 整个组共享同一套token，组内所有头保留相同的token集合

SnapKV为每个头独立选择token，而FastKV按头组统一选择token，并且通过TSP机制实现了预填充阶段的加速

TSP 如何选择重要token？

利用Attention机制本身来评估token的重要性。

  1. 计算注意力分数：在选定的TSP层，对于该层的每一个Attention Head，我们都能得到一个注意力分数矩阵 Att_l。
  2. 计算token重要性得分：为了得到一个统一的、代表整个层观点的分数，论文将所有Attention Head的分数进行平均： [ S_{i}^{TSPlayer} = {h=0}^{H-1} S{i}^{TSPlayer, h} ]
    • ( S_{i}^{TSPlayer} )：第 i 个token在TSP层的重要性得分。
    • ( H )：该层的Attention Head总数。
    • ( S_{i}^{TSPlayer, h} )：第 i 个token在第 h 个Head中的重要性得分（这个得分可以理解为对注意力分数进行滑窗平均得到的）。
  3. 选择Top-K：根据 ( S_{i}^{TSPlayer} ) 得分，选择得分最高的K个token。这个K就是 TSP长度（TSP Length）。论文发现TSP长度设为2048是个比较好的平衡点。
  • 注意一个时间节点：

也就是说TSP操作发生的时间点是“after the MLP block”，在 TSP 层，模型首先像正常一样，对所有输入 token 执行了 自注意力计算 和 MLP 计算。在经过了 MLP 块之后得到了该层最终的、经过充分信息处理的隐藏状态。此时，FastKV 才使用之前选出的重要 token 的索引，从这个完整的隐藏状态矩阵中，索引出那些重要 token 对应的隐藏状态。这些被挑出来的隐藏状态，会被传递给下一层。由于 token 数量大大减少，下一层的计算量也随之大幅降低。

两个关键参数的解耦

TSP长度 和 KV Cache预算（KV Budget） 是两个独立的参数。这样的涉及便于实际应用中平衡速度和精度。

  • TSP长度：决定有多少个token会参与TSP层之后的计算，影响预填充阶段的计算速度（TTFT）。
  • KV Budget：决定最终有多少个token的KV值会被存入Cache，影响生成阶段的内存占用和吞吐量（Throughput）。

例如，可以设置 TSP长度=2048，KV Budget=512。这意味着： - 预填充时，后期层只处理2048个token，计算量小，TTFT低。 - 生成时，KV Cache里只存了512个token的Key和Value，内存占用小，吞吐量高。

5. 一些疑问和思考

5.1 （续上面对SnapKV和FastKV的对比）如果有超长代码生成等任务（或者其他细粒度依赖的任务），FastKV采用的是按组进行token选择，而不是为每个注意力头单独选择token，假设在代码生成任务中，有一个注意力头专门负责结构体的定义，另一个头负责api的调用。这两个头关注的内容不同，因此它们认为重要的token也可能不同。如果这两个头被分在同一个组，那么组的重要性得分是这两个头得分的平均。这可能导致一些对其中一个头非常重要但对另一个头不重要的token被平均掉，从而不被选择。而在SnapKV中，每个头都可以保留自己最重要的token，因此不会出现这个问题。
